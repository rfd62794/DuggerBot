# ADR-008: Five-Level Agent Hierarchy and Twin Deference Model

## Status
Accepted

## Context

DuggerBot runs on two machines with different roles. Robert works primarily from
his laptop (Nitro 5). Claude connects to DuggerBot's MCP tools to extend its
capabilities. Multiple agents participate in both the development workflow and
the runtime execution chain.

Two problems required a formal decision:

1. **Authority ambiguity.** Without a defined hierarchy, it was unclear whether
   Tower or Nitro 5 should be the primary MCP connection point for Claude, and
   whether Tower should defer to the laptop or the reverse.

2. **API quota conflicts.** If Local TOBOR (Nitro 5) and Remote TOBOR (Tower)
   both route to the same provider simultaneously — Gemini Flash free tier, for
   example — they silently consume shared quota. RALPH on Tower and an on-demand
   Claude request on Nitro 5 hitting Gemini at the same time is a coordination
   failure, not a coincidence.

## Decision

### The Five-Level Hierarchy

Authority flows top to bottom. Each level receives direction from the level above
and may delegate downward. No level initiates contact with a level above it.

```
1. Robert          — Human. Absolute authority. Defines intent.
2. Claude          — Director/Architect. Receives Robert's intent. Directs work.
3. Devin           — Coding agent. Executes Claude's directives. MCP read access.
4. Local TOBOR     — Nitro 5 MCP server. Claude's primary tool layer when online.
5. Remote TOBOR    — Tower MCP server. Sub-agent. Background infrastructure.
```

Levels 1–3 are the development chain. Levels 4–5 are the runtime chain.
Devin (level 3) has read-only MCP access to Local TOBOR for build-time
verification — not for runtime task execution. See "Devin MCP Access" below.

### Twin Deference Rules

**When Nitro 5 is online:**
- Claude connects to Local TOBOR (Nitro 5 MCP, port 8001) as primary
- Local TOBOR handles all on-demand tool calls from Claude
- Local TOBOR reads Remote TOBOR's current provider usage via the Tower
  state REST API (OQ-003 resolution) before routing — prevents quota conflicts
- Local TOBOR may delegate overflow tasks to Remote TOBOR
- Local TOBOR enforces a configurable timeout on Remote TOBOR responses
  (see `DELEGATION_TIMEOUT_SECONDS` below). If Remote TOBOR does not respond
  within the timeout, Local TOBOR handles the task locally regardless of
  provider load. Tower's connectivity issues must never block Robert's session.

**When Nitro 5 is offline:**
- Remote TOBOR (Tower) accepts Claude connections directly as fallback
- Remote TOBOR routes all tasks locally with no coordination needed
- Tower's scheduled tasks (RALPH, morning dispatch) continue uninterrupted

**Remote TOBOR rules (always):**
- Handles tasks delegated from Local TOBOR
- Handles its own scheduled/background tasks (RALPH, morning dispatch,
  heartbeat polling)
- Never initiates contact with Claude
- Never claims to be the primary connection point while Nitro 5 is online
- Accepts Claude connections only in fallback mode (Nitro 5 offline)

**Scheduled tasks never defer:**
RALPH, morning dispatch, and all cron/heartbeat tasks remain on Tower
regardless of Nitro 5 presence. These are Tower's permanent responsibility.
Deference is for on-demand routing only.

### Provider Coordination Mechanism

Before Local TOBOR routes any on-demand task:
1. Request Tower's daily usage summary via `GET /twin/state/usage` 
2. If a provider is above 80% of its free tier daily limit on Tower, deprioritize
   it in Local TOBOR's routing chain for this request
3. Route to the least-loaded available provider
4. If the state request times out (`STATE_REQUEST_TIMEOUT_SECONDS`), proceed
   with default routing — coordination failure must not block Robert's session

This ensures Gemini Flash free tier quota is shared across both instances,
not consumed independently.

### Configurable Timeouts

Timeout values are environment variables, not locked constants. Tunable
per-instance without a superseding ADR.

| Variable | Default | Purpose |
|---|---|---|
| `DELEGATION_TIMEOUT_SECONDS` | `5` | Max wait for Remote TOBOR task response |
| `STATE_REQUEST_TIMEOUT_SECONDS` | `2` | Max wait for Tower usage state before routing blind |

Both variables go in `.env.example` and `config/instance.yaml`. Changing
default values requires updating both files and restarting the service.

### Nitro 5 Presence Protocol

Tower cannot reliably detect Nitro 5 (Tower has known connectivity issues).
Nitro 5 is always the initiator:

- Nitro 5 comes online → `POST Tower:/twin/register` 
- Nitro 5 sends heartbeat every 30 seconds → `POST Tower:/twin/heartbeat` 
- Tower marks Nitro 5 STALE after 3 missed heartbeats (90 seconds)
- Tower marks Nitro 5 OFFLINE after 5 missed heartbeats (150 seconds)
- Tower resumes full authority when Nitro 5 is OFFLINE
- Tower never polls for Nitro 5 — it only listens for registration

### Devin MCP Access

Devin (level 3) may connect to Local TOBOR's MCP server for build-time
verification. This is read-only, developer-specific access — not runtime
task execution.

**Auth model:** A separate `DEVIN_AUTH_TOKEN` identifies Devin callers.
The auth layer returns caller identity (not just pass/fail), enabling
handlers to apply per-caller routing rules.

**Two tool groups on the same MCP server:**

*Production tools* (Claude, and Devin in read-only mode):
- `research()`, `fast_lookup()`, `local_inference()` — when called by Devin,
  route to cheapest available option only. Never touch Claude API budget.
- `get_provider_status()`, `get_cost_today()` — unrestricted read access.

*Developer tools* (Devin only — not exposed to Claude):
- `get_project_state()` — reads `docs/state/current.md` 
- `get_open_issues()` — reads `docs/issues/` 
- `get_migration_manifest()` — reads `migration_manifest.json` 
- `verify_test_floor()` — runs pytest, returns count and pass/fail
- `check_coverage()` — returns per-module coverage percentages

Developer tools run on Local TOBOR only. Remote TOBOR does not expose them.
They are build-time tools scoped to active development sessions, not
background infrastructure.

**What this changes in Phase 2 (auth.py):**
`verify_token()` must return a `CallerIdentity` (CLAUDE or DEVIN) in addition
to passing/rejecting. Handlers use this to apply routing restrictions.
This is a small but real retroactive change to Phase 2 — file as ISSUE-002
to be resolved before Phase 3 certifies.

## Consequences

**This ADR locks:**
- Local TOBOR (Nitro 5) is Claude's primary runtime layer when online
- Remote TOBOR (Tower) is a sub-agent — never addresses Claude directly
  unless in fallback mode
- Scheduled tasks belong to Tower permanently and unconditionally
- Provider quota coordination is Local TOBOR's responsibility, not Tower's
- Devin has MCP access to Local TOBOR — read-only, developer tools only,
  production tools restricted to cheapest routing

**This ADR does NOT lock:**
- Timeout values — configurable via env vars, no ADR needed to change them
- The specific developer tool list — new developer tools may be added without
  a new ADR as long as they remain read-only

**This ADR defers:**
- OpenAgent as a shared MCP layer for Claude and Devin (separate project —
  named primitive, not in DuggerBot scope)
- DUGGERWORKSHOP as a third twin (Level 6 if the repo earns it)
- Separate TWIN_AUTH_TOKEN for inter-twin communication (currently uses
  MCP_AUTH_TOKEN — warranted in a future ADR if security posture requires it)

**Failure modes explicitly accepted:**
- If Tower's state REST is unreachable, Local TOBOR routes without coordination.
  Quota conflicts in this case are accepted as a known degraded mode.
- If Remote TOBOR is slow, the configurable timeout means the task runs locally
  at potentially higher provider cost. Accepted.
- If both instances hit Gemini simultaneously in the state-request window,
  they may double-consume quota. Coordination is best-effort, not guaranteed.
