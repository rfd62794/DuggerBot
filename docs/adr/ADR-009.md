# ADR-009: Version Tracking and Update Protocol

## Status
Accepted

## Context

DuggerBot runs as NSSM services on Tower (production) and potentially Nitro 5
(development). Changes made on Nitro 5 during active development need to propagate
to Tower reliably. NSSM's restart behavior, if misconfigured, can loop on a broken
update and brick a service. DuggerBot also cannot install itself on a machine it
has never run on — something external must do first-time setup.

Four problems required a formal decision:

1. **No version identity.** DuggerBot has no way to report what version it is,
   compare itself against its twin, or determine whether a restart is warranted.

2. **Update without self-destruction.** A running process cannot restart its own
   NSSM service — the call kills the caller. The restart must be delegated to
   NSSM via a clean exit, but only under controlled conditions.

3. **Update during active work is destructive.** If Remote TOBOR restarts while
   Local TOBOR is waiting on a delegation response, the delegation times out and
   the task fails. The update check must be aware of in-flight work.

4. **NSSM misconfiguration silently bricks the service.** Restart-on-any-exit
   without a delay or a failure ceiling will rapid-fire restart on a broken git
   pull indefinitely. The correct NSSM config must be prescribed, not suggested.

## Decision

### Version Scheme

```
MAJOR.MINOR.PATCH.rN
```

Where `N` = `git rev-list --count HEAD` (total commit count on current branch).

Examples: `0.1.0.r174`, `0.2.0.r201`, `1.0.0.r350`

**The `r` suffix is a deployment coordinate, not a semantic signal.**

`0.1.0.r174` and `0.1.0.r175` may contain breaking changes between them.
Semantic meaning lives exclusively in `MAJOR.MINOR.PATCH` and follows standard
SemVer rules. The revision tells you WHERE you are in commit history — it does
not tell you WHAT changed. Never infer compatibility from the `r` suffix alone.

Revision is computed at runtime:

```python
import subprocess

def get_current_revision() -> int:
    result = subprocess.run(
        ["git", "rev-list", "--count", "HEAD"],
        capture_output=True, text=True
    )
    return int(result.stdout.strip()) if result.returncode == 0 else 0

def get_remote_revision() -> int:
    subprocess.run(["git", "fetch"], capture_output=True)
    result = subprocess.run(
        ["git", "rev-list", "--count", "origin/main"],
        capture_output=True, text=True
    )
    return int(result.stdout.strip()) if result.returncode == 0 else 0
```

Version string at startup: `f"0.1.0.r{get_current_revision()}"`

### Self-Update Mechanism

DuggerBot cannot restart its own NSSM service. It exits cleanly and NSSM
restarts it. The exit code is the signal:

| Exit Code | Meaning | NSSM Behaviour |
|---|---|---|
| `0` | Clean update — restart with new version | Restart after delay |
| `1` | Error — do not loop | Stop service |

**Update flow:**

```
1. Update check fires (scheduled or Telegram command)
2. git fetch → compare local revision vs origin/main revision
3. If remote_revision <= local_revision: no update, return
4. Check: is TwinCoordinator in-flight? → if yes, defer N minutes
5. Check: is RALPH cycle active? (Phase 4+) → if yes, defer N minutes
6. git pull → if pull fails: exit(1) → NSSM stops, human intervention
7. If pull succeeds: exit(0) → NSSM restarts with new version
```

**Exit 1 on git pull failure is mandatory.** A failed pull followed by exit 0
creates a restart loop: NSSM restarts, DuggerBot finds the same update, tries
to pull again, fails again, exits 0 again. Exit 1 on any pull failure breaks
the loop and requires human resolution.

### Update Check Deferral (Hard Requirement)

The update check must not proceed if work is actively in-flight.

**Phase 3 deferral condition:**
`TwinCoordinator` has an in-flight delegation — a `DelegationRequest` has been
sent and no `DelegationResponse` has been received yet.

**Phase 4 deferral condition (added when RALPH lands):**
RALPH research loop is in an active research cycle — a task has been dequeued
from `HEARTBEAT.md` and has not yet written its output.

When deferred: schedule a retry after `UPDATE_CHECK_RETRY_MINUTES` (default: 5).
Do not block the event loop waiting. Use `asyncio.sleep` in a background task.

The coordinator's `DELEGATION_TIMEOUT_SECONDS` is a safety net — it means a
stuck delegation eventually clears. But relying on the timeout to unblock an
update is not acceptable. The check must actively defer, not passively race.

### NSSM Prescriptive Configuration

These settings are **required**, not optional. The Initializer script sets them.
Deviation from these values without a superseding ADR is a misconfiguration.

```powershell
# Set via nssm or the Initializer script — both Tower and Nitro 5
nssm set DuggerBot AppExit default Restart   # Exit 0 → restart
nssm set DuggerBot AppExit 1 Stop            # Exit 1 (error) → stop, no loop
nssm set DuggerBot AppRestartDelay 5000      # 5 second delay before restart
nssm set DuggerBot AppThrottle 10000         # Additional throttle between rapid restarts
nssm set DuggerBot AppStopMethodConsole 1500 # 1.5s graceful shutdown before kill
nssm set DuggerBot AppStopMethodWindow 1500
nssm set DuggerBot AppStopMethodThreads 1500
```

The throttle means: if DuggerBot exits within 10 seconds of starting, NSSM adds
an additional 10 second delay before the next restart. This prevents rapid-fire
restart loops from a bad update that exits immediately.

**What each setting protects against:**

| Setting | Protects against |
|---|---|
| `AppExit 1 Stop` | Loop on broken git pull |
| `AppRestartDelay 5000` | Immediate restart before filesystem settles |
| `AppThrottle 10000` | Crash-loop on a bad build |
| Graceful shutdown timers | Async task teardown — gives FastAPI time to close |

### DuggerBot Cannot Self-Initialize

DuggerBot cannot install itself on a machine it has never run on. The Initializer
script (`scripts/initialize.ps1`) handles first-time setup and is the canonical
way to deploy DuggerBot on any new instance. It is also idempotent — safe to run
on an existing installation.

**Initializer responsibilities (in order):**

```
1. Verify prerequisites: git, uv, NSSM are installed
2. Clone repo if not present, or git pull if already present
3. Run: uv sync
4. Prompt for .env.local values (or accept --env-path argument)
5. Set SOUL_PATH — operator copies soul documents separately via Tailscale/RDP
6. Register NSSM service with prescriptive config above
7. Start service: nssm start DuggerBot
8. Verify: poll /health until 200 or timeout (30s)
```

The Initializer does NOT copy soul documents — those are private, handled
out-of-band by the operator per ADR-008. The Initializer sets the SOUL_PATH
env var. Filling that path is the operator's responsibility.

### Twin Upgrade Notification

`POST /twin/upgrade` — already exists on the twin router (Phase 3).

The notification is **advisory only.** The receiving instance:
1. Checks its own in-flight state (same deferral rules as above)
2. If clear: fetches and compares revision
3. If update available: pulls and exits 0
4. If deferred: returns `{"accepted": false, "reason": "in-flight work", "retry_after_minutes": N}`

Neither instance forces the other to restart. Neither instance restarts on the
other's behalf. Each instance owns its own lifecycle.

## Consequences

**This ADR locks:**
- Version format: `MAJOR.MINOR.PATCH.rN` — permanent
- Revision source: `git rev-list --count HEAD` on current branch
- Exit 0 = clean restart, exit 1 = error stop — NSSM interprets these
- NSSM settings are prescriptive — deviation is a misconfiguration
- Update check must defer during in-flight delegation (Phase 3+) and active RALPH cycles (Phase 4+)
- DuggerBot never self-initializes — the Initializer script is the only bootstrap path
- Twin upgrade notification is advisory — each instance owns its own restart

**This ADR does NOT lock:**
- `UPDATE_CHECK_RETRY_MINUTES` — configurable via env var, default 5
- Update check schedule interval — configurable via env var
- Whether Nitro 5 runs NSSM or a shell watchdog — operator decision per instance

**Failure modes explicitly accepted:**
- If git pull fails on Tower, the service stops and requires human intervention.
  This is preferable to a restart loop. Accepted.
- If both instances detect an update simultaneously and both pull at the same time,
  they may briefly be at different revisions during the restart window. Accepted —
  the twin protocol is designed to tolerate short offline periods.
- If the update check defers indefinitely because RALPH never idles, the update
  never applies automatically. The Telegram command for manual update check exists
  to break this condition. Accepted.
